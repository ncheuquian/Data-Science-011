_A_r_g_u_m_e_n_t_s:

   topic: usually, a name or character string specifying the topic for
          which help is sought.  A character string (enclosed in
          explicit single or double quotes) is always taken as naming a
          topic.

          If the value of ‚Äòtopic‚Äô is a length-one character vector the
          topic is taken to be the value of the only element.
          Otherwise ‚Äòtopic‚Äô must be a name or a reserved word (if
          syntactically valid) or character string.

          See ‚ÄòDetails‚Äô for what happens if this is omitted.

 package: a name or character vector giving the packages to look into
          for documentation, or ‚ÄòNULL‚Äô.  By default, all packages whose
          namespaces are loaded are used.  To avoid a name being
          deparsed use e.g.  ‚Äò(pkg_ref)‚Äô (see the examples).

 lib.loc: a character vector of directory names of R libraries, or
          ‚ÄòNULL‚Äô.  The default value of ‚ÄòNULL‚Äô corresponds to all
          libraries currently known.  If the default is used, the
          loaded packages are searched before the libraries.  This is
          not used for HTML help (see ‚ÄòDetails‚Äô).

 verbose: logical; if ‚ÄòTRUE‚Äô, the file name is reported.

try.all.packages: logical; see ‚ÄòNote‚Äô.

help_type: character string: the type of help required.  Possible
          values are ‚Äò"text"‚Äô, ‚Äò"html"‚Äô and ‚Äò"pdf"‚Äô.  Case is ignored,
          and partial matching is allowed.

     The following types of help are available:

        ‚Ä¢ Plain text help

        ‚Ä¢ HTML help pages with hyperlinks to other topics, shown in a
          browser by ‚ÄòbrowseURL‚Äô.

          (On Unix-alikes, where possible an existing browser window is
          re-used: the macOS GUI uses its own browser window.)

          If for some reason HTML help is unavailable (see
          ‚ÄòstartDynamicHelp‚Äô), plain text help will be used instead.

        ‚Ä¢ For ‚Äòhelp‚Äô only, typeset as PDF - see the section on ‚ÄòOffline
          help‚Äô.

     On Unix-alikes: The ‚Äòfactory-fresh‚Äô default is text help except
          from the macOS GUI, which uses HTML help displayed in its own
          browser window.

     On Windows: The default for the type of help is selected when R is
          installed - the ‚Äòfactory-fresh‚Äô default is HTML help.

     The rendering of text help will use directional quotes in suitable
     locales (UTF-8 and single-byte Windows locales): sometimes the
     fonts used do not support these quotes so this can be turned off
     by setting ‚Äòoptions(useFancyQuotes = FALSE)‚Äô.

     ‚Äòtopic‚Äô is not optional: if it is omitted R will give

        ‚Ä¢ If a package is specified, (text or, in interactive use only,
          HTML) information on the package, including hints/links to
          suitable help topics.

        ‚Ä¢ If ‚Äòlib.loc‚Äô only is specified, a (text) list of available
          packages.

        ‚Ä¢ Help on ‚Äòhelp‚Äô itself if none of the first three arguments is
          specified.

     Some topics need to be quoted (by backticks) or given as a
     character string.  These include those which cannot syntactically
     appear on their own such as unary and binary operators, ‚Äòfunction‚Äô
     and control-flow reserved words (including ‚Äòif‚Äô, ‚Äòelse‚Äô ‚Äòfor‚Äô,
     ‚Äòin‚Äô, ‚Äòrepeat‚Äô, ‚Äòwhile‚Äô, ‚Äòbreak‚Äô and ‚Äònext‚Äô).  The other
     ‚Äòreserved‚Äô words can be used as if they were names, for example
     ‚ÄòTRUE‚Äô, ‚ÄòNA‚Äô and ‚ÄòInf‚Äô.

     If multiple help files matching ‚Äòtopic‚Äô are found, in interactive
     use a menu is presented for the user to choose one: in batch use
     the first on the search path is used.  (For HTML help the menu
     will be an HTML page, otherwise a graphical menu if possible if
     ‚ÄògetOption("menu.graphics")‚Äô is true, the default.)

     Note that HTML help does not make use of ‚Äòlib.loc‚Äô: it will always
     look first in the loaded packages and then along ‚Äò.libPaths()‚Äô.

_O_f_f_l_i_n_e _h_e_l_p:

     Typeset documentation is produced by running the LaTeX version of
     the help page through ‚Äòpdflatex‚Äô: this will produce a PDF file.

     The appearance of the output can be customized through a file
     ‚ÄòRhelp.cfg‚Äô somewhere in your LaTeX search path: this will be
     input as a LaTeX style file after ‚ÄòRd.sty‚Äô.  Some environment
     variables are consulted, notably ‚ÄòR_PAPERSIZE‚Äô (_via_
     ‚ÄògetOption("papersize")‚Äô) and ‚ÄòR_RD4PDF‚Äô (see ‚ÄòMaking manuals‚Äô in
     the ‚ÄòR Installation and Administration‚Äô manual).

     If there is a function ‚Äòoffline_help_helper‚Äô in the workspace or
     further down the search path it is used to do the typesetting,
     otherwise the function of that name in the ‚Äòutils‚Äô namespace (to
     which the first paragraph applies).  It should accept at least two
     arguments, the name of the LaTeX file to be typeset and the type
     (which is nowadays ignored).  It accepts a third argument,
     ‚Äòtexinputs‚Äô, which will give the graphics path when the help
     document contains figures, and will otherwise not be supplied.

_N_o_t_e:

     Unless ‚Äòlib.loc‚Äô is specified explicitly, the loaded packages are
     searched before those in the specified libraries.  This ensures
     that if a library is loaded from a library not in the known
     library trees, then the help from the loaded library is used.  If
     ‚Äòlib.loc‚Äô is specified explicitly, the loaded packages are _not_
     searched.

     If this search fails and argument ‚Äòtry.all.packages‚Äô is ‚ÄòTRUE‚Äô and
     neither ‚Äòpackages‚Äô nor ‚Äòlib.loc‚Äô is specified, then all the
     packages in the known library trees are searched for help on
     ‚Äòtopic‚Äô and a list of (any) packages where help may be found is
     displayed (with hyperlinks for ‚Äòhelp_type = "html"‚Äô).  *NB:*
     searching all packages can be slow, especially the first time
     (caching of files by the OS can expedite subsequent searches
     dramatically).

     require(stats); require(graphics)
     par(mfrow = c(2, 2))
     plot(Nile)
     acf(Nile)
     pacf(Nile)
     ar(Nile) # selects order 2
     cpgram(ar(Nile)$resid)
     par(mfrow = c(1, 1))
     arima(Nile, c(2, 0, 0))
     
     ## Now consider missing values, following Durbin & Koopman
     NileNA <- Nile
     NileNA[c(21:40, 61:80)] <- NA
     arima(NileNA, c(2, 0, 0))
     plot(NileNA)
     pred <-
        predict(arima(window(NileNA, 1871, 1890), c(2, 0, 0)), n.ahead = 20)
     lines(pred$pred, lty = 3, col = "red")
     lines(pred$pred + 2*pred$se, lty = 2, col = "blue")
     lines(pred$pred - 2*pred$se, lty = 2, col = "blue")
     pred <-
        predict(arima(window(NileNA, 1871, 1930), c(2, 0, 0)), n.ahead = 20)
     lines(pred$pred, lty = 3, col = "red")
     lines(pred$pred + 2*pred$se, lty = 2, col = "blue")
     lines(pred$pred - 2*pred$se, lty = 2, col = "blue")
     
     ## Structural time series models
     par(mfrow = c(3, 1))
     plot(Nile)
     ## local level model
     (fit <- StructTS(Nile, type = "level"))
     lines(fitted(fit), lty = 2)              # contemporaneous smoothing
     lines(tsSmooth(fit), lty = 2, col = 4)   # fixed-interval smoothing
     plot(residuals(fit)); abline(h = 0, lty = 3)
     ## local trend model
     (fit2 <- StructTS(Nile, type = "trend")) ## constant trend fitted
     pred <- predict(fit, n.ahead = 30)
     ## with 50% confidence interval
     ts.plot(Nile, pred$pred,
             pred$pred + 0.67*pred$se, pred$pred -0.67*pred$se)
     
     ## Now consider missing values

     (fit3 <- StructTS(NileNA, type = "level"))
     lines(fitted(fit3), lty = 2)
     lines(tsSmooth(fit3), lty = 3)
     plot(residuals(fit3)); abline(h = 0, lty = 3)
